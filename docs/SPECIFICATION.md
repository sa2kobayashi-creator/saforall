## 1. はじめに

### 1.1 目的

本書は、AI コードエディタ **saforall** が満たすべき機能要件・非機能要件・制約を定義する。実装・設計・テストの判断基準とする。

### 1.2 背景

開発者はコード編集と AI 支援を別ツールで行き来することが多い。saforall は、ローカルワークスペース上での編集と AI 対話を同一デスクトップアプリに統合し、Cursor のような体験を目指す。

### 1.3 スコープ

| 含む | 含まない（当面） |
| --- | --- |
| デスクトップアプリ（Windows 優先） | 本番向けクラウド SaaS |
| ローカルファイルの閲覧・編集・保存 | クラウド同期・共同編集（当面） |
| AI チャット / コード編集支援 | 独自 LLM の学習・ホスティング |
| XAMPP（Apache + MySQL）によるローカル API / 永続化 | VS Code 拡張互換マーケット |
| ワークスペース単位のコンテキスト利用 | |

### 1.4 用語

| 用語 | 定義 |
| --- | --- |
| ワークスペース | ユーザーが開いたローカルフォルダをルートとする作業単位 |
| アクティブファイル | エディタで現在表示・編集中のファイル |
| コンテキスト | AI に渡すコード・パス・選択範囲などの情報 |
| エージェント | ツール（読取・書込・検索等）を使いタスクを進める AI モード |
| IPC | Electron メインプロセスとレンダラ間の通信 |
| バックエンド API | XAMPP の Apache 上で動く PHP REST API |
| XAMPP | ローカル開発用の Apache / MySQL / PHP 環境 |

## 2. 製品概要

### 2.1 ビジョン

ローカルコードベースを理解し、説明・修正・実装を対話で進められる AI ネイティブなコードエディタ。

### 2.2 ターゲットユーザー

- 個人開発者・小規模チーム
- TypeScript / JavaScript を中心に、一般的な言語も扱う開発者
- Cursor / VS Code + AI 拡張に近い体験を求めるユーザー

### 2.3 利用環境

| 項目 | 要件 |
| --- | --- |
| OS | Windows 10/11（優先） |
| クライアント | Electron（Chromium + Node.js） |
| バックエンド | XAMPP（Apache + PHP + MySQL） |
| 既定 URL | `http://localhost/saforall/`（開発時、設定変更可） |
| ネットワーク | AI 機能利用時は外部 LLM API への接続が必要 |
| 認証情報 | API キーは設定として保存（MySQL またはローカル設定。生値の扱い方針は設計書参照） |

### 2.4 システム役割分担

| 層 | 技術 | 役割 |
| --- | --- | --- |
| クライアント | Electron + React + Monaco | 編集 UI、ローカルファイル操作、ユーザー操作 |
| Web サーバ | Apache（XAMPP） | PHP API の配信 |
| API | PHP | 設定・会話履歴・メタデータの CRUD、将来の AI プロキシ |
| DB | MySQL（XAMPP） | 永続データの保存 |

## 4. 画面・操作仕様

### 4.1 画面構成

```
┌──────┬──────────┬────────────────────┬────────────┐
│ Act. │ Sidebar  │ Editor             │ AI Chat    │
│ Bar  │ Explorer │ (Monaco)           │ Panel      │
│      │          │                    │            │
├──────┴──────────┴────────────────────┴────────────┤
│ Status Bar                                        │
└───────────────────────────────────────────────────┘
```

| 領域 | 役割 |
| --- | --- |
| Activity Bar | ワークスペース操作、チャット表示切替 |
| Sidebar | ファイル一覧・選択 |
| Editor | コード表示・編集・保存 |
| AI Chat | 対話入力・履歴表示 |
| Status Bar | パス・保存状態などの短い状態表示 |

### 4.2 主要ユースケース

#### UC-01: ワークスペースを開いてファイルを編集する

1. ユーザーが「フォルダを開く」を実行する
2. OS のフォルダ選択ダイアログが表示される
3. 選択したフォルダがワークスペースになり、一覧が表示される
4. ファイルを選択するとエディタに内容が表示される
5. 編集後、保存操作でディスクに書き込まれる

#### UC-02: AI にコードについて質問する（目標）

1. ユーザーがファイルを開いた状態でチャットに質問を入力する
2. Electron がコンテキスト付きでバックエンド API（またはメイン経由）へ送信する
3. PHP が会話を MySQL に保存し、LLM API を呼び出す
4. 回答がチャットに表示される
5. 必要に応じて提案コードをローカルファイルへ適用する

#### UC-03: バックエンドの疎通確認

1. XAMPP で Apache と MySQL を起動する
2. `GET /api/health` が `ok` と DB 接続成功を返す
3. Electron 起動時または設定画面で接続状態を確認できる

### 4.3 キーボードショートカット（初期）

| 操作 | キー |
| --- | --- |
| 保存 | Ctrl + S / Cmd + S |
| チャット送信 | Enter（Shift+Enter で改行） |

## 5. 非機能要件

### 5.1 性能

| ID | 要件 |
| --- | --- |
| NF-P01 | 起動後、空のワークスペース状態までおおむね数秒以内に操作可能であること |
| NF-P02 | 1 MB 程度のテキストファイルを遅延なく開いて編集できること |
| NF-P03 | AI 応答はチャンク単位で逐次表示し、完了まで UI をブロックしないこと |

### 5.2 信頼性

| ID | 要件 |
| --- | --- |
| NF-R01 | ファイル保存失敗時はエラーを表示し、バッファ内容を失わないこと |
| NF-R02 | LLM API 失敗時はユーザー向けメッセージを表示し、再送可能であること |
| NF-R03 | 未保存ファイルがある場合、終了時に確認する（P1） |

### 5.3 セキュリティ

| ID | 要件 |
| --- | --- |
| NF-S01 | レンダラで `nodeIntegration` を無効化し、`contextIsolation` を有効にする |
| NF-S02 | ファイルシステム操作は preload 経由の限定 API のみとする |
| NF-S03 | API キーをリポジトリ・ログ・レンダラのグローバル変数に露出しない |
| NF-S04 | ワークスペース外パスへの意図しない書込を防ぐ（パス検証） |
| NF-S05 | DB パスワード・API キーを含む設定ファイルを Git に含めない |
| NF-S06 | バックエンド API は開発時は localhost のみを想定する（本番公開は別設計） |

### 5.4 保守性・移植性

| ID | 要件 |
| --- | --- |
| NF-M01 | クライアントは TypeScript による型付けを必須とする |
| NF-M02 | UI・ローカル fs・バックエンド API の責務をモジュール境界で分離する |
| NF-M03 | Windows + XAMPP を第一ターゲットとする |
| NF-M04 | PHP API はシンプルな構成とし、フレームワーク依存を最小にする（初期） |

### 5.5 ユーザビリティ

| ID | 要件 |
| --- | --- |
| NF-U01 | ダークテーマを既定とする（VS Code / Cursor に近い配色） |
| NF-U02 | UI 文言は日本語を既定とする |

## 6. 外部インタフェース

### 6.1 バックエンド API（Apache / PHP）

- ベース URL（開発既定）: `http://localhost/saforall/api`
- 形式: JSON（`Content-Type: application/json`）
- 文字コード: UTF-8

| メソッド | パス | 説明 | 優先度 |
| --- | --- | --- | --- |
| GET | `/health` | 生存確認・DB 接続確認 | P0 |
| GET/PUT | `/settings` | アプリ設定 | P1 |
| GET/POST | `/chat/sessions` | 会話セッション一覧・作成 | P1 |
| GET/POST | `/chat/sessions/{id}/messages` | メッセージ取得・追加 | P1 |
| GET/POST | `/workspaces` | 最近のワークスペース | P1 |
| POST | `/ai/chat` | LLM プロキシ（将来） | P1 |

詳細なリクエスト/レスポンス形は設計書を参照する。

### 6.2 LLM API

- OpenAI 互換 API を第一候補とする
- 呼び出しは **PHP バックエンド経由**を基本とする（API キーをサーバ側で保持）
- 開発初期の暫定として Electron メインから直接呼ぶ選択肢は残すが、最終形は BE-09 に寄せる

### 6.3 MySQL

- XAMPP 付属の MySQL / MariaDB
- データベース名（既定）: `saforall`
- 接続ユーザー（開発既定）: `root`（パスワードはローカル設定）

### 6.4 ローカルファイルシステム

- 読取・書込・ディレクトリ一覧・メタ情報取得は **Electron メインプロセス**が担当（PHP はソースコード本体を扱わない）
- 文字コードは当面 UTF-8 を前提とする

### 6.5 OS ダイアログ

- フォルダ選択ダイアログ（Electron `dialog`）

## 7. データ要件

| データ | 保存場所 | 備考 |
| --- | --- | --- |
| 編集中バッファ | メモリ（レンダラ） | 保存操作でローカルディスクへ |
| ソースコード本体 | ローカルファイルシステム | MySQL には保存しない |
| アプリ設定 | MySQL `settings` | モデル名・base URL 等 |

| 会話履歴 | MySQL `chat_sessions` / `chat_messages` | |
| 最近のワークスペース | MySQL `workspaces` | パス文字列のみ |
| DB 接続情報 | `server/config/database.php` | Git 管理外 |

## 8. 制約・前提

1. 商用 LLM の利用料金・利用規約はユーザー責任とする
2. バイナリファイルの編集は対象外（テキスト前提）
3. 巨大リポジトリ（数万ファイル超）の全件インデックスは P2 以降
4. **Apache / MySQL が停止していても**、ローカル編集機能は利用可能とする（OT-03）
5. バックエンドは当面 **localhost 専用**（XAMPP 開発環境）とする
6. ソースコードの実体は常にローカルディスク上にあり、DB はメタデータと会話のみを持つ
